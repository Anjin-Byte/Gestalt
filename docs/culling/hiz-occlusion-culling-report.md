## 1. Executive Summary

This codebase already has enough upstream spatial structure to support practical GPU occlusion culling, but it does not yet expose the two core runtime primitives required by a Hi-Z pipeline:

1. An explicit depth texture/pyramid build path in the renderer backend.
2. A first-class cull-unit bounds buffer (AABB/sphere), preferably at brick/chunk granularity.

Targeting clarification:

1. The long-term and universal target is greedy-meshed chunk rendering (`docs/greedy-meshing-docs/voxel-mesh-architecture.md`).
2. Current concrete runtime evidence in this repository still comes from the active voxel/instanced path (`apps/web/src/viewer/outputs.ts`, `apps/web/src/modules/wasmVoxelizer/runCore.ts`).
3. This report treats the current path as transitional and maps it to the greedy-mesh target design.

Current status by topic:

1. Depth source: produced internally by Three renderer draw, but not owned as an explicit app-managed target in `apps/web/src/viewer/threeBackend.ts:112`.
2. Bounds source (viewer): per-voxel positions and instance transforms are built in `apps/web/src/viewer/outputs.ts:88` and `apps/web/src/viewer/outputs.ts:169`, but no dedicated cull-unit metadata buffer is exported.
3. Better granularity (voxelizer): brick/chunk structure already exists in `apps/web/src/modules/wasmVoxelizer/runCore.ts:124`, and brick bounds are already iterated in `apps/web/src/modules/wasmVoxelizer/runCore.ts:133`.

Bottom line: the shortest path is to establish a chunk-level culling contract compatible with greedy-mesh outputs, seed it from current brick/chunk data, and run per-frame Hi-Z build + GPU cull in backend render flow.

## 4. Depth Buffer for Hi-Z Pyramid

### 4.1 What exists now

The backend render implementation currently issues a high-level renderer draw:

- `renderer?.render(scene, camera)` at `apps/web/src/viewer/threeBackend.ts:114`.

Depth is generated during this draw, but your code does not currently:

1. Bind an explicit depth texture target under app control.
2. Preserve that depth target as an input to a downsample pass.
3. Build and store a depth mip pyramid.

### 4.2 Why this is a blocker for Hi-Z

Hi-Z culling requires a sampleable, multi-resolution depth hierarchy for conservative occlusion tests. If depth only exists as an internal transient attachment in an opaque render path, the app cannot reliably:

1. Downsample it to mip levels.
2. Bind it in a compute culling pass.
3. Synchronize pass order and resource lifetime explicitly.

### 4.3 What must be added

At minimum, backend render flow needs:

1. Explicit depth prepass or explicit depth capture target.
2. A pyramid build pass (downsample depth -> mip 1..N).
3. A culling pass that consumes the pyramid before final draw.

Proposed ownership location:

1. Add a Hi-Z subsystem under viewer/backend (for example `apps/web/src/viewer/hiz/`).
2. Initialize GPU resources when WebGPU backend is active.
3. Execute per-frame at/near `apps/web/src/viewer/threeBackend.ts:112`.

### 4.4 Compatibility note

You already access the raw GPU device for diagnostics via `renderer.backend.device` in `apps/web/src/viewer/webgpuDiagnostics.ts:12`. This proves the backend can reach low-level WebGPU internals, but that alone is insufficient; Hi-Z still needs an explicit render-pass/resource plan rather than passive diagnostics access.

## 5. Transitional Occludee Bounds Evidence (`outputs.ts`)

### 5.1 Data currently present

In the current voxel-instanced output build:

1. Raw voxel positions are read from module output in `apps/web/src/viewer/outputs.ts:88`.
2. Instanced transforms are authored with `instanced.setMatrixAt(...)` at `apps/web/src/viewer/outputs.ts:169`.
3. Built-in frustum culling is disabled (`instanced.frustumCulled = false`) at `apps/web/src/viewer/outputs.ts:159`.

### 5.2 What this implies for the greedy-mesh target

You can reconstruct bounds from existing data, but today they are not represented as a durable cull buffer. Specifically:

1. There is no per-cull-unit AABB/sphere array stored for GPU culling.
2. There is no visibility bitmask/indirect args buffer driven by bounds tests.
3. Culling granularity is tied to the current render object shape, not a dedicated chunk visibility system.

### 5.3 Why this is transitional only

Per-voxel culling has poor cost/benefit in most scenes and is not aligned with the universal greedy-mesh architecture:

1. Too many cull tests and metadata entries.
2. High CPU/GPU overhead for visibility bookkeeping.
3. Better throughput comes from chunk-oriented cull units that match greedy-mesh ownership and update boundaries.

## 6. Bridge Source: Brick/Chunk Bounds in `runCore.ts`

### 6.1 Existing data path

In paged sparse mode:

1. Brick records are flattened in `apps/web/src/modules/wasmVoxelizer/runCore.ts:124`.
2. Active page bricks are selected and expanded in `apps/web/src/modules/wasmVoxelizer/runCore.ts:130`.
3. Brick bounds are already iterated for debug line geometry in `apps/web/src/modules/wasmVoxelizer/runCore.ts:133`.

### 6.2 Why this is ideal as a migration bridge

Brick/chunk units are already:

1. Spatially coherent.
2. Close to the data production stage.
3. Significantly fewer than per-voxel instances.
4. Large enough to amortize cull cost, but small enough to reduce overdraw materially.

### 6.3 Immediate opportunity for greedy-mesh alignment

The current debug bounds iteration can be promoted to production metadata emission, then mapped to greedy chunk mesh cull units:

1. Compute AABB center/extents (or sphere center/radius) per active brick/chunk.
2. Emit this as structured metadata in module output.
3. Provide a deterministic mapping from brick-derived clusters to final greedy chunk meshes.
4. Feed metadata directly into GPU culling without re-deriving in viewer layer.

## 7. Greedy-Mesh Culling Strategy (Target State)

Hi-Z should be modeled around chunk mesh rendering, not voxel instance rendering:

1. Primary cull unit: one greedy-mesh chunk render object.
2. Optional finer unit: sub-chunk cluster bounds only when chunk meshes become large/heterogeneous.
3. Occluders: opaque greedy chunk meshes in depth prepass.
4. Occludees: chunk mesh bounds tested against Hi-Z pyramid.

Alignment with existing design docs:

1. Stable render objects and geometry swaps are already defined in `docs/greedy-meshing-docs/threejs-buffer-management.md`.
2. Chunk lifecycle and ownership are defined in `docs/greedy-meshing-docs/module-system-lifecycle.md`.
3. Chunk-first architecture is defined in `docs/greedy-meshing-docs/voxel-mesh-architecture.md`.

Implication:

1. The culling data contract should be owned by the greedy-mesh module outputs.
2. Current voxelizer brick bounds should be treated as temporary source data while migrating.

## 9. Proposed Data Model Changes

### 9.1 Module output extension (recommended)

Add a culling descriptor that is shared by greedy-mesh module outputs, with optional temporary support for voxel outputs during migration:

```ts
type CullingDescriptor = {
  // one entry per cull unit
  centers: Float32Array; // xyz packed
  extents: Float32Array; // xyz packed (AABB half-size)
  // or: radii: Float32Array
  count: number;
  // maps clusters to render units (chunk mesh index, draw range, etc.)
  firstRenderUnit: Uint32Array;
  renderUnitCount: Uint32Array;
};

type MeshDescriptor = {
  positions: Float32Array;
  indices?: Uint32Array;
  normals?: Float32Array;
  colors?: Float32Array;
  cull?: CullingDescriptor; // primary target: greedy chunk meshes
};

type VoxelsDescriptor = {
  positions: Float32Array<ArrayBufferLike>;
  voxelSize: number;
  cull?: CullingDescriptor; // transitional bridge only
};
```

Generation source:

1. Final target: emit chunk (or chunk-subcluster) bounds from greedy-mesh modules.
2. Migration bridge: derive culling metadata from `runCore.ts` brick/chunk data.
3. Fallback: viewer-derived coarse bounds when metadata is missing.

### 9.2 Normalization and ownership

Decide once whether bounds are:

1. World-space static values (recommended for current module architecture).
2. Object-space values with transform per frame (better for animated objects).

For current and planned chunk flows (mostly static output snapshots with versioned updates), world-space bounds are simpler.

## 10. Proposed Frame Pipeline (WebGPU path)

Recommended per-frame sequence:

1. Update controls/camera.
2. Depth prepass for opaque greedy chunk meshes to explicit depth texture.
3. Build Hi-Z depth pyramid (mip downsample).
4. Run compute culling for chunk (or chunk-subcluster) bounds.
5. Compact visible draws (or write visibility mask).
6. Execute main color pass using visible greedy chunk mesh set only.

For WebGL fallback:

1. Keep current path unchanged initially.
2. Optionally add CPU frustum/portal culling only.

## 11. Algorithm and Resource Notes

### 11.1 Depth pyramid convention

Choose and document one depth convention across passes:

1. Standard depth (near=0, far=1) with conservative max/min rule per mip.
2. Reversed-Z if adopted later, with opposite reduction rule.

A mismatch here causes false-visible or false-occluded bugs.

### 11.2 Bounds test shape

Practical hierarchy:

1. Sphere test for speed (fast reject/accept).
2. Optional AABB refinement for tighter cull.

For first implementation, sphere-only is acceptable if cull units are small (brick-level).

### 11.3 Memory budgeting

Depth pyramid cost is predictable:

1. Base depth level + geometric mip chain (~1.33x base texel count).
2. Cluster bounds buffers scale linearly with cluster count.
3. Visibility + compaction buffers add fixed overhead per frame.

## 12. Implementation Plan (Incremental)

### Phase 1: Data contract and metadata

1. Extend module output types with optional cull metadata in `apps/web/src/modules/types.ts`.
2. Define greedy-mesh module output metadata for chunk cull units (primary contract).
3. Use `apps/web/src/modules/wasmVoxelizer/runCore.ts` brick bounds as migration bridge data.
4. Preserve metadata through `ModuleHost` -> `Viewer` -> backend.

Exit criteria:

1. Debug overlay can report cluster count and bounds count.
2. Metadata survives activation/re-run paths.
3. Same schema works for both bridge data and greedy-mesh outputs.

### Phase 2: Backend depth target + pyramid

1. Add explicit depth target management in viewer backend WebGPU branch.
2. Add mip pyramid textures and downsample compute pipeline.
3. Validate pyramid correctness with debug visualizations.

Exit criteria:

1. Stable depth pyramid generated every frame at locked/unlocked resolutions.

### Phase 3: GPU culling and draw integration

1. Build culling compute pass consuming camera matrices + bounds + pyramid.
2. Produce visibility mask or compacted indirect args.
3. Route greedy chunk mesh draws to visible subset only.

Exit criteria:

1. Overdraw drops in occluded scenes.
2. No visible popping under moderate camera movement.

### Phase 4: Robustness and fallback

1. Add toggles/metrics in debug panel.
2. Add safe fallback to current render path on capability failure.
3. Add regression tests for correctness and performance envelopes.

## 13. Risks and Mitigations

### Risk 1: Over-culling due to depth convention mismatch

Mitigation:

1. Lock depth convention in docs and shaders.
2. Add validation scenes with known occlusion outcomes.

### Risk 2: Culling cost exceeds benefit on small scenes

Mitigation:

1. Add threshold to bypass Hi-Z below N clusters.
2. Keep mode toggle in debug panel for A/B testing.

### Risk 3: Three.js backend integration complexity

Mitigation:

1. Isolate Hi-Z resources/pass orchestration in a backend-local subsystem.
2. Keep fallback path and fail-open behavior.

### Risk 4: Inconsistent metadata across bridge modes and final greedy modules

Mitigation:

1. Define one canonical cluster schema.
2. Populate it consistently in bridge outputs and final greedy-mesh module outputs.

## 14. Validation Plan

Correctness checks:

1. Occluded clusters never render when fully hidden.
2. Visible clusters remain stable across camera movement.
3. No temporal flicker at cull boundaries.

Performance checks:

1. GPU time of cull+pyramid is lower than saved fragment/vertex cost in heavy scenes.

Instrumentation suggestions:

1. Add counters for tested clusters, visible clusters, and rejected clusters.
